# VelNit Life

**Relationship Intelligence for Life's Second Chapter.**

VelNit Life is an AI-powered Relationship Intelligence Platform helping
couples, families, caregivers and communities stay emotionally connected
through life's second chapter. This repository currently contains two
phases:

- **Phase 1** — the public marketing website and its core architecture.
- **Phase B** — the VelNit Content Operating System: a founder-only,
  review-first workspace (`/studio`) that turns a content idea into a
  brief, generates four draft formats grounded in VelNit's own Brain
  documents, and requires explicit founder approval before anything is
  considered ready to publish. **Nothing generated by the Content OS
  publishes automatically** - see "The Content Operating System" below.

Still out of scope: the authenticated member-facing SaaS dashboard, the
VelNit Companion AI product surface, healthcare workflows, and any real
publishing integrations (Facebook, Canva, HeyGen, email delivery) - those
remain planned for later phases.

## What's in Phase 1

- Marketing pages: Home, About, Relationship Intelligence, VelNit Care,
  VelNit Connect, VelNit Companion, Academy, Blog, Contact
- Beta registration and newsletter signup, backed by Supabase
- A Supabase-backed blog (categories, tags, author, search, pagination,
  Markdown content, related posts)
- Responsive navigation and footer
- A clean-architecture, Supabase-ready data layer
- Netlify deployment configuration
- SEO (metadata, sitemap, robots, JSON-LD, Open Graph image), accessibility
  (skip link, semantic landmarks, focus states, labelled forms) and
  performance foundations (self-hosted variable fonts, `next/image`, ISR
  on the blog)

## What's in Phase B — the Content Operating System

A founder-only workspace at `/studio` (gated by Supabase Auth plus
membership in a `founders` allowlist table - see
[docs/SUPABASE_SETUP.md](./docs/SUPABASE_SETUP.md)) that proves the
VelNit Brain (VRIF, TALK, Writing DNA, Knowledge Graph - see `brain/`) can
generate consistent, on-brand content before any automatic publishing is
built:

- **Ideas** (`/studio`) - capture a topic and audience.
- **Briefs** (`/studio/ideas/[id]`) - turn an idea into a structured brief:
  primary emotion, desired outcome, TALK stage, VRIF pillars, a practical
  action, a call to action, prohibited claims, and linked knowledge
  references.
- **Generation** (`/studio/briefs/[id]`) - one click generates four draft
  formats from the same brief: a long-form article, a Facebook post, an
  email newsletter, and a 45-60 second reel script. Every draft records
  which knowledge references and prompt version produced it, and a
  per-format failure never blocks the other three.
- **Review** (`/studio/review`) - approve, request revision, edit
  manually, or add review notes, checked against the same ten-point
  checklist the model is asked to self-check against
  (`lib/ai/review-checklist.ts`).
- **No automatic publishing.** Facebook, Canva, HeyGen, and newsletter
  delivery are not connected. Approving a draft only creates a
  `publication_jobs` stub row for future phases - nothing in this
  codebase advances it further.

Content moves through nine statuses end to end: `idea` → `brief_ready` →
`generating` → `draft` → `needs_revision` (as many times as needed) →
`approved` → `scheduled` → `published` → `archived` - the last two are
reserved for a future phase with real publishing integrations.

## Tech stack

| Layer | Choice |
|---|---|
| Framework | Next.js 15 (App Router), React 19, TypeScript |
| Styling | Tailwind CSS, hand-rolled shadcn/ui-style primitives, Framer Motion |
| Data | Supabase (Postgres + Row Level Security) |
| Forms | React Hook Form + Zod |
| Email | Resend (optional, transactional only) |
| AI (Content OS) | Anthropic Messages API, behind a swappable provider interface (`lib/ai/provider.ts`) |
| Testing | Vitest |
| Hosting | Netlify (`@netlify/plugin-nextjs`) |

## Getting started

```bash
npm install
cp .env.example .env.local   # then fill in your Supabase project details
npm run dev
```

Visit `http://localhost:3000`. See
[docs/SUPABASE_SETUP.md](./docs/SUPABASE_SETUP.md) to provision the
database (including the Phase B Content Operating System schema and how
to add yourself as a founder), and
[docs/ENVIRONMENT_VARIABLES.md](./docs/ENVIRONMENT_VARIABLES.md) for a
full list of variables. The public site builds and runs even without
Supabase configured — forms and the blog simply degrade gracefully.
`/studio` requires Supabase (it is founder-only, nothing in it is public)
but works fully without an AI provider configured - only draft generation
needs `ANTHROPIC_API_KEY` / `ANTHROPIC_MODEL`.

### Scripts

```bash
npm run dev              # start the dev server
```

## Project structure

```
app/            Route segments (App Router) - marketing pages, plus /studio (Phase B)
brain/          Source-of-truth VelNit Brain docs: VRIF, TALK, Writing DNA, Knowledge Graph
components/     Shared UI: ui/ primitives, layout/ (nav, footer), marketing/
features/       Feature-scoped client code: beta, newsletter, contact, blog, studio
hooks/          Shared React hooks
lib/            Cross-cutting utilities: supabase clients, validation, SEO, constants, auth, ai
scripts/        Build-time codegen (compiles brain/*.md into lib/ai/brain-content.generated.ts)
services/       Server-side data access: blog reads, Content OS reads/writes
supabase/       SQL migrations and seed data
tests/          Vitest unit tests (validation, services, auth, AI review checklist)
types/          Shared TypeScript types, including the Supabase `Database` type
content/        Reserved for future static/MDX content
docs/           Project documentation (this folder)
```

This follows clean-architecture separation: `app/` owns routing and
composition only; `features/` owns client-side form state and Server
Actions; `services/` owns Supabase reads/writes used by Server Components
and Server Actions; `lib/` has no dependency on React. UI primitives in
`components/ui` are intentionally dependency-free and reusable across any
feature.

## Data layer

Supabase is the system of record for beta signups, newsletter
subscribers, contact messages, blog content, and the Content Operating
System. See:

- `supabase/migrations/0001_init.sql` — Phase 1 schema + Row Level Security
- `supabase/migrations/0002_content_os.sql` — Phase B schema: `founders`,
  `content_ideas`, `content_briefs`, `content_items`, `content_reviews`,
  `knowledge_references`, `publication_jobs`, all founder-only under RLS
- `supabase/seed/seed.sql` — optional demo blog content
- `supabase/seed/content_os_seed.sql` — optional pilot idea/brief and
  knowledge references for the Content OS
- `types/database.ts` — hand-written types mirroring both schemas (see
  the file header for how to regenerate these from the Supabase CLI)
- `lib/supabase/` — four clients: `client.ts` (browser), `server.ts`
  (cookie-aware, for Server Components/Actions - the only one that can
  satisfy Content OS RLS policies), `public.ts` (cookie-free, for
  statically generated/ISR blog pages), and `admin.ts` (service-role,
  reserved for future admin tooling)

Public forms use React Hook Form + Zod for validation and call Server
Actions in `features/*/actions.ts`, which insert directly into Supabase
under the public Row Level Security policies — no service-role key is
required for Phase 1 to be fully functional. Content OS actions in
`features/studio/*.ts` follow the same Server Action + Zod pattern, but
every one starts with `requireFounder()` (`lib/auth/founder.ts`) and reads
and writes through founder-only RLS policies instead.

## The VelNit Brain

`brain/` holds the four foundational documents that govern every word the
Content OS generates - VRIF (theory), the TALK Model (practice), Writing
DNA (voice), and the Knowledge Graph (structure/memory) - plus
`CHANGELOG.md` and `DECISIONS.md`, the version-control files the
Knowledge Graph document itself calls for. These are the single source of
truth: `npm run generate:brain` compiles them into
`lib/ai/brain-content.generated.ts`, which `lib/ai/prompts/v1.ts` sends as
system context on every generation call. Edit the markdown, regenerate,
commit both - never hand-edit the generated file. See `brain/README.md`.

## Testing

```bash
npm run test
```

Vitest covers content creation (Zod schema validation), review status
transitions (`setContentItemStatus`, `createContentReview` - including
that approval stamps `approved_at` and revision requests don't), and
authorization (`getFounder` / `requireFounder` against a mocked Supabase
client, covering no-session, signed-in-but-not-a-founder, and authorized
cases). Services and auth are tested against a lightweight fake Supabase
client (`tests/helpers/supabase-mock.ts`) rather than a live database, so
the suite runs in under a second with no network access.

## Deployment (Netlify)

This repo includes `netlify.toml`, pre-configured with the official
`@netlify/plugin-nextjs` build plugin, security headers, and a note on
which environment variables to set in the Netlify UI. Connect the repo to
Netlify, set the environment variables from
[docs/ENVIRONMENT_VARIABLES.md](./docs/ENVIRONMENT_VARIABLES.md), and
deploy — no further configuration is required. `npm run build` compiles
the Brain automatically via the `prebuild` script.
